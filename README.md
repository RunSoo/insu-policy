# Toss 스타일 보험금 심사 UI 개발 가이드 📘

이 프로젝트는 **'해외여행자 보험' 등의 사고 경위를 입력받고 AI가 약관을 분석해 지급/부지급 여부를 판정해 주는 인터페이스**를 구현한 데모입니다. 처음에는 단일 질문 구조에서 시작하여, 현재는 **연속적인 타임라인 뷰, 로컬스토리지 데이터 보존, 커스텀 스크롤 사이드바 내비게이션**까지 갖춘 완성도 높은 프로토타입으로 발전했습니다.

## 🛠 기술 스택 (Tech Stack)
- **Core**: React, TypeScript, Vite
- **Styling**: Tailwind CSS (v4)
- **Animation**: Framer Motion
- **Icons**: Lucide React
- **Utilities**: clsx, tailwind-merge

## 💻 로컬 실행 및 테스트 방법 (Getting Started)

이 프로젝트를 로컬 환경에서 실행하고 테스트하는 방법입니다.

### 1. 의존성 설치
프로젝트 폴더로 이동한 후, 패키지 매니저를 사용하여 의존성을 설치합니다.
```bash
# npm을 사용하는 경우
npm install

# yarn을 사용하는 경우
yarn install

# pnpm을 사용하는 경우
pnpm install
```

### 2. 로컬 개발 서버 실행
개발 서버를 실행하여 실시간으로 변경 사항을 확인하며 테스트할 수 있습니다.
```bash
npm run dev
# 또는 yarn dev / pnpm dev
```
- 실행이 완료되면 터미널에 표시되는 로컬 주소로 브라우저를 통해 접속합니다.
  - **사용자 화면 (보상가이드 메인)**: `http://localhost:5173/`
  - **관리자 화면 (약관 파싱 어드민)**: `http://localhost:5173/admin.html`

### 3. 프로덕션 빌드 및 프리뷰 (선택 사항)
실제 배포 환경과 동일하게 빌드하고 테스트하려면 다음 명령어를 사용합니다.
```bash
# 애플리케이션 빌드
npm run build

# 빌드 결과물 로컬 프리뷰
npm run preview
```

## 🏗 컴포넌트 아키텍처 및 동작 원리

전체 앱은 중앙의 `App.tsx`를 중심으로 여러 하위 컴포넌트가 유기적으로 결합하여 동작합니다.

### 1. `App.tsx` (메인 레이아웃 및 상태 관리)
- **역할**: 전체 타임라인 배열(`flow`)을 상태로 관리하는 최상위 부모 컴포넌트입니다.
- **동작 원리**:
  - `flow` 배열은 여러 개의 `FlowItem`(질문 폼 + 답변 결과) 객체로 구성됩니다.
  - 앱이 처음 마운트될 때 **로컬스토리지(`localStorage`)**에서 기존 데이터를 읽어와 복원합니다. 이때 작성된 지 24시간이 넘은 오래된 내역은 만료 처리되어 자동 청소됩니다.
  - `flow` 배열을 순회(`map`)하며, 각 항목의 상태에 따라 빈 폼이면 `InsuranceForm`을, 답변이 있다면 `AnalysisResult`를 함께 렌더링합니다.
  - 화면 상단에는 스크롤을 내려도 따라다니며 배경이 은은하게 비치는(Backdrop Blur) **Sticky Header**가, 우측 하단에는 **맨 아래로 스크롤(FAB)** 버튼이 위치하여 쾌적한 내비게이션을 돕습니다.

### 2. `InsuranceForm.tsx` (입력 폼 컴포넌트)
- **역할**: 사용자의 사고 경위와 보험 상품을 입력받습니다.
- **동작 원리**:
  - 제출 전에는 텍스트 에어리어와 드롭다운을 활성화하고 'AI 분석 시작' 버튼을 렌더링합니다. (개발용 테스트 강제 옵션 포함)
  - 폼이 제출되어 부모 컴포넌트에 의해 `disabled` 상태로 전환되면, 폼 조작이 비활성화(`pointer-events-none`)되고 하단 버튼 영역은 사라집니다. 
  - 대신 우측 하단에 **질문 등록 시각(연/월/일 포함)**이 깔끔한 텍스트로 남게 되어, 지나간 질문들이 하나의 잘 정돈된 '히스토리 기록'으로 변모합니다.

### 3. `AnalysisResult.tsx` (분석 결과 컴포넌트)
- **역할**: AI의 분석(또는 백엔드 통신) 과정을 시각화하고 최종 판정 결과를 렌더링합니다.
- **동작 원리**:
  - **스켈레톤 렌더링**: 초기에는 2.5초짜리 부드러운 스켈레톤 로딩(깜빡임)이 재생되어 사용자에게 진행 상황을 알려줍니다.
  - **로컬스토리지 연동**: 이미 부모로부터 전달받은 분석 결과(`initialResultData`)가 있다면 불필요한 지연 시간을 생략하고 곧바로 결과를 렌더링합니다.
  - **상태 편집 모드 (기능 보존)**: 분석 결과(지급/부지급)와 약관 내용을 사용자가 직접 수정할 수 있는 편집 모드가 코드 내부에 내장되어 있습니다. (현재 UI에서는 잠시 주석 처리해 가려둠)
  - **결과 표시**: 지급 가능(파란색), 거절(빨간색) 배지를 통해 직관적인 피드백을 제공하며, 약관 본문 내에 결과의 이유가 되는 텍스트를 노란색 형광펜(`bg-yellow-200`)으로 하이라이트(`highlight.tsx` 유틸리티 활용)합니다.

### 4. `Sidebar.tsx` (사이드바 내비게이션)
- **역할**: 과거에 질문했던 내역들을 한눈에 모아보고 원하는 곳으로 빠르게 점프할 수 있는 서랍장입니다.
- **동작 원리**:
  - 헤더의 햄버거 메뉴(☰)를 누르면 우측에서 `framer-motion`을 타고 부드럽게 튀어나옵니다. 뒷배경은 블러 처리되어 현재 화면에 깊이감을 더합니다.
  - 빈 폼을 제외한 과거의 질문들을 '보험 상품 배지 + 시간 + 사고 경위 두 줄 요약' 형태로 예쁘게 나열합니다.
  - 특정 항목을 클릭하면 사이드바가 스스로 닫히며 해당 질문의 위치(DOM ID)를 향해 스무스하게 스크롤업됩니다.

### 5. `scroll.ts` (커스텀 스크롤 물리 유틸리티)
- **역할**: 브라우저의 기본 `window.scrollTo`의 밋밋함을 대체하는 쫀득한 스크롤 기능을 제공합니다.
- **동작 원리**:
  - `framer-motion`의 `animate` 함수를 활용해 `window.scrollY` 값을 0.8~1.0초간 제어합니다.
  - `easeOutQuart` 느낌의 커스텀 베지어 곡선(`[0.25, 1, 0.5, 1]`)을 적용하여, **이동을 시작할 땐 빠르고 목적지에 다다를 땐 아주 부드럽게 감속**하는 소위 '촤라라락' 감기는 고급스러운 스크롤 모션을 만들어냅니다.

### 6. 어드민 콘솔 및 브라우저 자체 PDF 파싱 엔진 (`admin.html`, `pdfParser.ts`)
- **역할**: 관리자가 보험 약관 PDF를 업로드하면 브라우저 내에서 직접 마크다운으로 추출해주는 전용 백오피스입니다.
- **동작 원리**:
  - **Vite MPA (Multi-Page Application)** 설정을 통해, 메인 앱과 저장소 및 의존성을 공유하면서도 빌드 시에는 완전히 독립된 SPA(`admin.html`)로 분리되도록 아키텍처를 구성했습니다.
  - **`pdfjs-dist` 기반 휴리스틱 파싱 엔진**: 외부 백엔드 서버의 도움 없이 브라우저 단독으로 PDF를 파싱합니다. 텍스트 조각들의 Y/X 좌표를 추출 및 정렬하고, 텍스트 간의 간격 오차를 계산하여 표(Table) 영역을 스스로 유추하고 마크다운 테이블(`| 셀 |`) 형태로 변환해 내는 고도의 휴리스틱 알고리즘(`src/utils/pdfParser.ts`)이 내장되어 있습니다.

### 7. Dify API 연동 및 계층형 문서 관리
- **역할**: Dify(LLM 백엔드)에 등록된 지식 베이스(데이터셋)와 문서를 브라우저에서 직접 관리합니다.
- **동작 원리**:
  - **계층형 UI 구조**: `지식 베이스(폴더) 목록 -> 특정 지식 베이스 클릭 -> 내부 문서(파일) 목록 및 상태 조회 -> 새 문서 파싱 및 전송` 형태의 직관적인 드릴다운(Drill-down) UI를 구현했습니다.
  - **CORS 우회 및 환경 변수**: 로컬 개발 시 발생하는 CORS(교차 출처 리소스 공유) 에러를 방지하기 위해 `vite.config.ts`에 프록시(Proxy)를 설정하였으며, 보안을 위해 API 키와 타겟 URL은 `.env` 환경 변수로 격리하여 관리합니다.

## 🚀 개발 및 진화 과정 (Step-by-Step)

### Phase 1: 기초 구조 및 디자인 시스템
- **Vite + React + TS** 환경 구축.
- Tailwind CSS v4를 설정하고, 토스 스타일의 **둥근 모서리(`rounded-[32px]`), 넉넉한 여백, 부드러운 그림자** 등 기본 토큰을 정립했습니다.
- 약관 텍스트에서 키워드를 찾아 하이라이트하는 정규식 기반 유틸리티를 완성했습니다.

### Phase 2: 연속 질문 타임라인 (Timeline Flow)
- 한 번 묻고 끝나는 구조에서 벗어나, 대화형 피드처럼 아래로 계속 폼이 추가되는 **배열 기반 타임라인(`flow`)**으로 개편했습니다.
- '다른 질문하기'를 누르면 기존 내역은 위쪽에 남고, 새 폼이 화면 맨 아래에 스르륵 등장합니다.

### Phase 3: 데이터 보존 (LocalStorage) 및 UI 디테일
- 사용자가 새로고침을 해도 질문과 답변이 유지되도록 브라우저의 `localStorage`와 연동했습니다. (보존 기간 24시간)
- 지나간 폼에 남아있던 거슬리는 제출 버튼들을 싹 지우고, 대신 '연월일/시간'이 기재된 타임스탬프를 폼과 결과 카드 하단에 통일감 있게 배치했습니다.
- 화면 최하단에 작고 연한 텍스트로 법적 면책 조항(서버 미저장 안내)을 추가했습니다.

### Phase 4: 쫀득한 모션과 사이드바 내비게이션
- 긴 타임라인을 제어하기 위해, **'맨 아래로 이동' 버튼**과 **'사이드바 리스트'**를 추가했습니다.
- 밋밋한 스크롤을 버리고 물리 기반의 **커스텀 애니메이션 스크롤**을 전면 도입하여 앱의 UX 완성도와 브랜드 경험을 극대화했습니다.

### Phase 5: 어드민 콘솔 추가 및 브라우저 자체 파싱 (MPA)
- 기존 서비스와 라우팅이 엉키지 않도록 **Vite MPA(다중 엔트리)** 구조를 도입하여 `admin.html`을 분리했습니다.
- 외부 서버 연동 없이 브라우저에서 곧바로 PDF 약관을 읽고 파싱하는 `pdfjs-dist` 기반 TS 엔진을 이식하였으며, 파이썬 기반 서버와 맞먹는 수준으로 텍스트 위치 좌표를 계산해 표(Table)와 조항(`# 제 n 조`)을 감지해 마크다운으로 변환합니다.
- 사용자 화면(메인 앱) 또한 토스 감성의 여백 중심 디자인과 **Pretendard** 폰트를 전역에 적용해 디자인 완성도를 극대화했습니다.

### Phase 6: Dify API 연동 및 계층형 지식 베이스 관리
- 어드민 페이지에 Dify API를 연동하여 실제 서버의 데이터셋(지식 베이스) 및 문서를 관리할 수 있는 기능을 추가했습니다.
- 로컬 환경의 CORS 에러를 우회하기 위해 Vite Proxy를 설정하고 환경 변수(`.env`) 아키텍처를 도입했습니다.
- UI 흐름을 '단일 파싱 화면'에서 **'지식 베이스 목록 ➡️ 내부 문서 목록 ➡️ 새 문서 파싱 및 업로드'**의 계층형 폴더 구조로 전면 개편하여 확장성과 사용성을 높였습니다.

## 💡 주요 디자인 패턴 요약
- **직관적인 컬러 피드백**: 긍정(지급)은 파란색, 부정(거절/보류)은 빨간색으로 철저히 통일했습니다.
- **연속적인 시각 트랜지션**: `AnimatePresence`를 통해 요소가 생겨나고 사라지는 모든 순간(스켈레톤->카드 전환, 사이드바 개폐, 새 폼 등장)에 끊김 없는 페이드 인/아웃 모션을 적용했습니다.
- **모바일 친화적 반응형**: 최대 너비(`max-w-4xl`)를 제한하여 PC에서도 콘텐츠가 흩어지지 않게 잡고, 모바일 기기에서도 좌우 패딩이 일정하게 유지되도록 설계했습니다.
